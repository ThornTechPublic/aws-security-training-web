Module 1: Create Users and Groups

Objectives

  • Create an IAM user account for each member of your team.
  • Add your IAM users to a group
  • Grant permissions by adding attaching policies to the Group

Overview

In this lab, you will continue setting up your AWS account.

The person logged in with the SSO account will provision an IAM account for each member of the group, as well as their own. The SSO account will not be used in any of the labs (but will still be available if you need it for some reason).

The SSO user will first create a Group. Permissions will then be assigned to this Group. These permissions are fairly liberal, and will give you the ability to perform the labs.

Then the SSO user will create an IAM user for each member of the team. Each user will be added to the Group. The SSO user will also relay the credentials as each user is created — just remember to check the box to require a password reset.

Create a Group with the System Administrator Managed Policy

  • Go to IAM

  • On the left, click on Groups

  • Click Create New Group

  • For the Group Name, type LabGroup

  • Click Next Step

  • On the Attach Policy page, check the box next to SystemAdministrator

  • Click Next Step

  • Click Create Group

Attach an inline policy

In this section, you're going to add an Inline policy. This will cover additional permissions you will need that are not included in the SystemAdministrator managed policy.

  • In IAM, click on LabGroup

  • Click on the Permissions tab

  • Under Inline Policies, expand the disclosure triangle

  • Click on the text click here to create a new inline policy

  • On the Set Permissions page, choose Custom Policy

  • Click Select

  • On the Review Policy page, enter a Policy Name of LabGroupPolicy

  • For the Policy document, paste in the following JSON text:

    {
  "Version": "2012-10-17",
  "Statement": [{
      "Effect": "Allow",
      "Action": [
        "cloudformation:*",
        "iam:*",
        "inspector:*",
        "ssm:*",
        "dynamodb:*",
        "rds:*",
        "cloudfront:*",
        "apigateway:*",
        "lambda:*",
        "waf:*"
      ],
      "Resource": "*"
  }]
}

  • Click Apply Policy

Create a User

In this section, you are going to create your IAM users.

If you are the one logged in with the SSO account, gather your other team members so that they're ready to receive their new credentials.

  • On the left, click on Users

  • Click Add user

  • Enter the user name

    • Click Add another user to create multiple users at once

  • Check both boxes for Programmatic and Console access

  • Keep the default options:

    • Autogenerated password
    • Require password reset

  • Click Next: Permissions

  • On the Set permissions page, check the box next to LabGroup

  • Click Next: Tags

  • Click Next: Review

  • Click Create users

    You should see the following screen:

  • Click the Download .csv button. This will download a credentials.csv file to your machine.

  • Relay the account credentials to your team members.

  • In the Success message at the top, you will see the AWS Management Console sign-in URL for your AWS account. Send this to your team members as well.

Testing Credentials as a team member

In this section, each member of your team will log into AWS using your IAM credentials.

  • Navigate to the AWS Management Console sign-in URL provided by your SSO administrator

  • Enter the username and password provided to you

  • You will be prompted to change your password on first login

Things to note

IAM vs SSO login

Most AWS environments use IAM accounts rather than SSO. Our use of SSO is primarily to facility this lab environment. In future labs, you will be using your IAM account.

Console vs Programmatic access

In this lab, you are granting your team members both Console and Programmatic access. Console is the web interface for AWS. And programmatic is a set of access keys that you include when you make CLI and API calls.

As a best practice, you can give users one or the other (or both) based on least privilege.

In the lab, if you're not comfortable with your SSO administrator seeing your Secret access key, feel free to rovision yourself a new one after login.

AWS Managed vs Customer Managed vs Inline policies

AWS Managed policies are useful for quickly granting broad permissions. You do not have the ability to edit AWS managed policies, because they are owned by AWS. Some AWS Managed policies (e.g. AmazonEC2RoleforSSM) are very useful because they pull together a set of distributed but commonly used permissions. Other AWS Managed policies (e.g. AdministratorAccess) should be used with caution.

Customer Managed policies are defined by you. These can be re-used for different identities.

Inline policies apply to a single identity, and are often used for ad hoc troubleshooting.

AWS Management Console Sign-In URL

If you ever need to remember the URL to the AWS console, go to the IAM Dashboard. Toward the top, you'll see the URL. You can also click Customize to make this URL more user friendly.
