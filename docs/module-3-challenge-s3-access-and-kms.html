Module 3 Challenge: S3 Access and KMS

Objectives

  • Use KMS to restrict S3 access at the resource level

Overview

You can use IAM policies and S3 bucket policies to restrict access to S3. But how do you prevent privileged users like system administrators from accessing salary and credit card data?

One approach is to use KMS encryption. While sysadmins may have full S3 access to handle operations, they won't necessarily be on the whitelist of users that can decrypt using KMS.

You can also use KMS to help safeguard against data leaks through S3 misconfiguration. One reason for the S3 data leaks in the news is that S3 has the option to make the data public because of its website feature. And whenever there's a wrong answer choice, someone is bound to pick it. Encrypting objects with KMS gives you a redundant safety layer (i.e. belts and suspenders), because anonymous users can't decrypt using KMS.

Challenge

In this challenge, you will encrypt an S3 object with KMS. Then you need to demonstrate that a team member with system administrator access cannot read the encrypted S3 object. If a team member is not available to help test, you can instead make the S3 bucket public, and demonstrate that an anonymous visitor cannot read the encrypted S3 object.

As with the other challenge, you will be given high level instructions, and it's up to you to poke around the AWS console on your own to accomplish these tasks. The hardest part of the challenge is finishing in the time remaining — if there is any — so keep an eye on the clock.

High level instructions

Create the KMS key

In this section, you need to create a KMS key. Each KMS key has two sets of permissions for who can (1) manage and (2) use it. You will create a key where you have exclusive access.

  • Search for KMS in the list of services.
  • Create a KMS key
  • Only allow your own IAM account to administer the key
  • Only allow your own IAM account to use the key

Encrypt an S3 object

Encryption on S3 is on a per object basis, and happens when the object is uploaded. The easiest way to encrypt an S3 object is to set the default encryption on the entire S3 bucket. This will encrypt newly uploaded objects, but not existing ones.

  • On your restricted S3 bucket, set the Default encryption to your KMS key
  • Use the S3 console to upload a new file
  • Verify that the uploaded file is encrypted with your KMS key

Testing system administrator access

To test whether KMS encryption is restricting access, ask a team member for help. If no team members are available to help, test this on your own by making the S3 bucket public.

  • Ask a colleague to access the encrypted S3 object. This should fail.
  • Ask a colleague to access an existing unencrypted file (i.e. restricted.txt) This should work.

Note: Using KMS to restrict access only works because IAM users are provisioned with the SystemAdministrator AWS Managed Policy. If you gave IAM users AdministratorAccess along the way for troubleshooting, they would have unrestricted access to all resources, including KMS.

Testing misconfigured public access

If a team member is unable to help test KMS access, you can instead make your S3 bucket public. Anonymous users should be able to see existing S3 objects, but get an error when trying to decrypt an object protected by KMS.

  • Make the contents of the restricted S3 bucket public, by applying the following S3 bucket policy (replace the bucket name below with your own):
{
  "Version":"2012-10-17",
  "Statement":[{
    "Effect":"Allow",
    "Principal": "*",
    "Action":["s3:GetObject"],
    "Resource":["arn:aws:s3:::example-bucket/*"]
  }]
}
  • Click the public URL of the unencrypted S3 object (you should be able to read it)
  • Click the public URL of the encrypted S3 object (you should get a KMS error)
